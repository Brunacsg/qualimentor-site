TDD, BDD e ATDD

  • TDD (Test Driven Development): Escreva testes antes do código. Quality Assurance participa criando testes de aceitação.
  • BDD (Behavior Driven Development): Cenários em linguagem natural. Gherkin para especificação executável.
  • ATDD (Acceptance Test Driven Development): Testes de aceitação definem requisitos.

Quality Assurance em DevOps

  • Shift Left Testing: Testes desde o início do desenvolvimento.
  • Continuous Testing: Testes automatizados em pipelines CI/CD.
  • Infrastructure as Code: Testes de infraestrutura e configurações.
  • Monitoring: Observabilidade e testes em produção.

Exemplo prático: Daily Standup

Formato de Daily Standup

João (Dev): "Ontem implementei o endpoint de login. Hoje vou trabalhar na validação. Sem bloqueios."

Maria (Quality Assurance): "Ontem testei a funcionalidade de cadastro. Hoje vou criar casos de teste para login. Bloqueada esperando deploy."

Pedro (PO): "Ontem revisei requisitos. Hoje vou priorizar backlog. Sem bloqueios."

Duração: 15 minutos máximo

Exemplo: Definition of Done

DoD para User Story

  • Código implementado e revisado
  • Testes unitários criados (>80% cobertura)
  • Testes de integração passando
  • Validação manual executada sem defeitos críticos
  • Documentação atualizada
  • Aprovado pelo Product Owner
  • Deployed em staging sem erros

Exemplo: BDD com Gherkin

User Story em Gherkin

Feature: Login de usuário
  Como usuário registrado
  Quero fazer login no sistema
  Para acessar minhas informações

  Scenario: Login com credenciais válidas
    Given que estou na página de login
    When preencho email "usuario@teste.com"
    And preencho senha "senha123"
    And clico em "Entrar"
    Then devo ver o dashboard
    And devo estar logado no sistema

  Scenario: Login com senha inválida
    Given que estou na página de login
    When preencho email válido
    And preencho senha incorreta
    And clico em "Entrar"
    Then devo ver mensagem "Senha inválida"

Exemplo: Three Amigos

Reunião Three Amigos

Product Owner: "Esta feature precisa validar CPF e permitir apenas maiores de 18 anos"

Developer: "Vou implementar validação no frontend e backend. Preciso de API de validação de CPF"

Quality Assurance: "Vou testar casos válidos/inválidos de CPF, idade, mensagens de erro. Preciso de dados de teste"

Resultado: Requisitos clarificados, riscos identificados, plano de teste definido
